# Agents 团队驾驶舱 (Agents Team Cockpit)

本项目要做的东西：一个盖在 GitHub Issues 之上的界面，让一套 AI agent 协作研发的工作流变得「看得见、能自动跑、能审架构」。本文件是这套语言的词汇表（glossary），只定义概念，不放实现细节。

## Language

**卡片 (Card)**:
一个 GitHub Issue 在驾驶舱里的呈现。
_Avoid_: ticket、任务（当指代不清时）

**流水线 (Pipeline)**:
卡片从讨论到交付要经过的阶段序列。形状是**分叉**而非直线：一张 **PRD** 走前段（`讨论` → `拆票`），拆完即退居为「TB 卡的归属泳道」；由它派生的多张 **TB 卡** 各自走后段（`待开发` → … → `完成`）。两段在 `拆票` 处相接。
_Avoid_: 把它当作 PRD 卡与 TB 卡共用的一条单一直线

**阶段 (Stage)**:
流水线上的一个位置，等同于卡片当前的一种状态。本词指**真实阶段**（领域权威、较细，如 `开发–审查环`、`smoke`、`验收`）。
_Avoid_: 列(column)；把"状态(state)"当同义词随意混用；与 **派生阶段** 混为一谈

**派生阶段 (Derived Stage / 显示桶)**:
驾驶舱给卡片上色用的**粗投影**——由驾驶舱从「标签 + open/closed + 关联 PR 状态」自动算出的 5 个桶之一：待打理 / 待开发 / 开发审查中 / 待我验收 / 完成。它是 **阶段**（真实阶段）的**有损投影**：多个真实阶段可能塌缩进同一桶（如 `smoke` 与 `验收` 现都落「待我验收」）。
_Avoid_: 当成真实阶段的精确镜像——它**故意粗**（PRD #1「阶段模型刻意粗」）

**自动段 (Auto Stage)**:
完成标准能「不靠人验收」、因此可被自动触发与推进的阶段。

**人工闸门 (Human Gate)**:
完成标准必须由人来验收、不能自动推进的阶段。**是一道「当前还搬不掉」的关卡，不是永久事实**——随工具与信任增强，闸门会被逐步改造成 **自动段**；这正是压低 **HITL 占比** 的主战场。
_例_: grill（需求讨论）、拆票、前端体验、smoke 联调

**人在环 / HITL (Human-in-the-Loop)**:
流水线推进中**必须由维护者亲自动手**的那部分——即卡片停在 **人工闸门** 上、等人来推的环节。**「HITL 占比」是本产品要尽力压低的核心指标**：驾驶舱的终极目的就是尽可能释放 **Agent** 产能、把维护者从环里拿出来。注意：**工作流本身会随本工具的演进而改变**（闸门减少、自动段增多），故本词汇表描述的是「当前」流程，非定局。

**开发–审查环 (Dev–Review Loop)**:
开发 agent 提 PR、审查 agent review、开发 agent 拉取后再改，直到通过的循环——自动化的最小单元。

**验收 (Acceptance)**:
判定一张 **卡片** 的活「真的做完了、可以放行到 **完成**」的动作。**能否自动，取决于"做完"是否机器可判**：纯逻辑卡 + 行为测试 + CI 绿 → **自动验收**（自动段）；需肉眼/联调判断 → 落到 **smoke**（人工闸门）。
_Avoid_: 与代码"对不对"（**开发–审查环** + 测试的事）混为一谈——验收问的是"可以放行了吗"

**smoke (烟测 / 联调验收)**:
**验收** 的**人工闸门**形态：维护者亲自把东西跑起来、用眼睛或手感确认它真的成立。用于"做完"机器判不了的卡（UI/体验、端到端联调），对齐「前端体验是人工闸门」。
_Avoid_: 当成自动化测试的同义词——smoke 是"人来拍板放行"，不是机器跑的测试

**Agent (选手)**:
一个扮演某角色干活的 AI 实例（一个 Claude Code 或 Codex 进程）。

**角色 (Role)**:
一个 Agent 在某个阶段承担的职责（开发 / 审查 / ……）。

**依赖边 (Dependency Edge)**:
两张 **卡片**（或卡片与 ADR/PR）之间的有向关系，从 issue 正文的 `#引用` + 关键词解析得到。
_例_: A `Depends on` B、`Source: ADR-0039`、标题里的 `TB1 → TB2`

**血脉 / 依赖簇 (Dependency Cluster)**:
一张 **卡片** 顺着 **依赖边**（**不分上下游方向**）连通到的所有卡片的集合——即图论上的「连通分量」。回答「点这张卡，牵一发动哪些卡」：含上游挡它的、下游被它挡的、以及「同被某张卡挡住」的难兄难弟。**跟随当前可见范围计算**：默认只含 open 卡；打开「显示 closed」后，已完成的卡与边才并入。是单击高亮、「只看我盯的」过滤所聚焦的单位。
_Avoid_: 与「只顺一个方向的上游链」或「PRD 同组」混为一谈——血脉是**无向连通**，不看方向、不看 PRD 归属

**PRD**:
一种 **卡片**（GitHub issue），描述一个要做的东西的需求面；是 `to-prd` 的产物、`to-issues` 的源。**住在 GitHub 上。**

**TB 卡 (Tracer-Bullet Card / 实现卡)**:
一种 **卡片**（GitHub issue），是 `to-issues` 从一张 **PRD** 拆出的、可独立被 **Agent** 领取的纵向切片实现卡（如本项目的 `TB1-1`…`TB1-7`）。是 **开发–审查环** 的作用对象。
_Avoid_: 与 **PRD** 混用——PRD 说「要做什么」，TB 卡是「拆出来去做」的一刀

**ADR**:
一篇**仓库内文档**（`docs/adr/NNNN-slug.md`），记录一个架构决策的「为什么」。**不是卡片，住在 repo 的文件里。**

**就绪 (Ready)**:
一种**派生状态**——一张 **卡片** 的 **阶段** 处于「待开发」(标签 `ready-for-agent`)且其所有 **依赖边**（`Blocked by`）指向的卡片都已关闭；即「现在就能放给 Agent 干」。是依赖 DAG 视图的高亮重点。

**为什么链 (Why-Chain)**:
一张 **卡片** 通往它的 **ADR** / **PRD** 的链路，回答「这张卡为什么存在、依据是什么」。
注意两端住在两个地方：**PRD 在 GitHub（卡片）**，**ADR 在 repo 文件（`docs/adr/`）**——驾驶舱得读两个数据源才能连起这条线。

**龄期 (Staleness / 停滞时长)**:
一张 **open 卡片** 距**上次活动**到现在的时长——即「这张卡没人碰、停了多久」。是 **HITL** 的一个负向信号：龄期越长，越可能是一张卡在 **人工闸门** 上、等维护者来推的卡。驾驶舱只把它作为 open 卡的停滞信号展示；closed 卡可保留时间信息用于追溯，但不显示「停滞」。与 **就绪** 互为对仗（就绪＝现在能往前推的正信号，龄期＝可能卡住了的负信号），两者都是驾驶舱从数据**算出来的派生信号**。
_Avoid_: 与「创建到现在多久」混为一谈——龄期问的是「多久没动了」（看上次活动），不是「这卡活了多久」（看创建时间）

**盯 / 重点 (Pin)**:
维护者给一张 **卡片** 打的**纯本地、二元**标记——「这张我盯着」。只存在本机（不写回 GitHub，故不触碰 ADR-0001 的只读边界），用于在视图里强调被盯的卡、并支持「只看我盯的」过滤。是这套信号里**唯一由人手动打的主观标记**，区别于 **就绪**、**龄期** 那种由驾驶舱从数据算出来的**派生信号**。
_Avoid_: 叫它「优先级」——它不分高/中/低档，只是「盯 / 不盯」的开关，将来若真要做分级优先级是另一个概念

## Relationships

- 一条 **流水线** 由若干 **阶段** 组成；每个 **阶段** 要么是 **自动段**，要么是 **人工闸门**
- **流水线** 在 `拆票` 处**分叉**：一张 **PRD** 走完「讨论 → 拆票」后退为归属泳道，其派生的多张 **TB 卡** 各自走「待开发 → … → 完成」
- **开发–审查环** 是流水线中段的一个 **自动段**，由「开发」「审查」两个 **角色** 的 **Agent** 来回接力
- 一张 **卡片** 在任一时刻停在某一个 **阶段** 上
- 能否自动化 = 该 **阶段** 的完成能否「不靠人验收」（这条原则区分了 **自动段** 与 **人工闸门**）
- 一张 **卡片** 的 **验收** 走 **自动段** 还是 **人工闸门（smoke）**，由「这张卡的'做完'是否**机器可判**」决定——这是压低 **HITL 占比** 的关键判据

## Example dialogue

> **我:** 「卡片到了『审查』这一站，能自动往前推吗？」
> **驾驶舱:** 「能——『审查』是 **自动段**，审查 Agent 通过后就进 **开发–审查环** 的下一圈或直接放行；但再往后的 smoke 联调是 **人工闸门**，得你亲自验收才动。」

## Flagged ambiguities

- 产品自身的名字：你最初叫它「工作台」，我提议「驾驶舱」。两者并存待定——**驾驶舱**强调「看得见+操控」，**工作台**强调「干活的地方」。下一步择一。
- 「状态」一词被用来既指 GitHub 上的 issue 字段、又指流水线 **阶段**；本项目统一用 **阶段** 指后者。
- 「看板」曾指 GitHub Projects 看板；现已弃用（见 ADR-0001）。本项目里「看板/视图」一律指**驾驶舱自己渲染**的图状视图。
