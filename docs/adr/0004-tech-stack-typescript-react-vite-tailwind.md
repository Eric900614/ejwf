# 驾驶舱技术栈：TypeScript + React/Vite + Tailwind 设计令牌 + cytoscape/dagre + Vitest

## 决定

TB1（以及由它立下基线的后续卡）采用以下技术栈，于 2026-06-03 与维护者 triage #2 时敲定：

- **语言 / 运行时**：**TypeScript + Node.js**。拉数据脚本、纯逻辑模块、画图前端，全程一套语言。
- **「本地 app」形态**：**Vite 构建的 React 单页应用（SPA）**，开发时 `npm run dev` 起本地开发服务、浏览器打开；可 `npm run build` 出一份纯本地静态产物。数据由一个 **Node 脚本调 `gh` 拉取后生成数据文件**供应用读取（沿用原型已验证的「确定性拉取 + 解析」路径）。
  Vite 的本地开发服务天然就是「本地小服务」形态，也为将来 Phase 2/3（拉本地 agent、读本地 repo 文件、页内一键刷新）预留了后端落点。
- **前端地基（在搭骨架时一次性夯实，避免样式返工）**：
  - **React 组件结构**：界面拆成清晰的组件层级（图容器 / 节点详情 / 工具条 / 图例 / linter 面板 等），纯逻辑与渲染解耦。
  - **设计令牌体系（Tailwind）**：把 **ADR-0003 已定的视觉语义**编码为设计令牌——5 阶段配色（待打理 / 待开发 / 开发审查中 / 待我验收 / 完成）、就绪态的「圈」语义（**金圈=就绪可派 / 绿圈=依赖已就位 / 有入箭头=仍被挡 / 无圈=本无依赖**），以及间距 / 字阶 / 圆角等基础令牌。**单一来源**：图（cytoscape 的样式）与界面（Tailwind 的 class）从同一套令牌取色，确保图与 UI 视觉一致。
  - **组件层**：约定「基础原子组件 → 组合业务组件」的分层，建立可复用约定（非铺满整套组件库——只建结构与约定）。
  - **响应式规则**：定下断点与布局行为约定（图区自适应、面板在窄屏的收纳方式）。
- **DAG 渲染**：**cytoscape.js + dagre 布局**（经 `cytoscape-dagre` 适配器），原型已验证可用；以 npm 依赖引入，挂载在一个 React 组件内（用 ref/effect 管理生命周期）。其节点样式从上面的设计令牌取色。
- **依赖解析**：**确定性正则，不上 LLM**（写卡约定高度结构化，正则即可高精度、可重复、可调试；对齐 PRD #1 与 ADR-0001）。
- **测试框架**：**Vitest**。纯逻辑深模块（PRD #1 的模块 3 关系解析器 / 4 阶段推导器 / 5 图构建+就绪）用「输入 → 断言输出」的行为测试；本卡确立这一测试范式（`ejwf` 原为空 repo）。
- **读 GitHub `main` 上的 ADR 版本**（对应将来的卡 #5 / ADR-0002）：倾向用 `gh api` 读取（与拉 issue 共用一套 `gh` 鉴权），**细节留到 #5 决定**，TB1 不涉及。

## 背景 / 为什么

PRD #1 与 ADR-0002 把技术栈「留给实现阶段决定」。原型（`prototype-dag/`）已用 `foravia` 真实数据跑通并经维护者认可，趟出了「TS/Node + cytoscape/dagre + 确定性解析」这条核心路线；本 ADR 把它确认为基线。

与原型「无框架、纯静态页」相比，本决定**额外引入 React + Vite + Tailwind**，原因是维护者明确要求：**在搭骨架阶段就建立 React 组件结构、视觉系统、Tailwind 设计令牌、组件层与响应式规则，以避免日后样式框架返工**。这一诉求与 #2 的定位吻合——#2 本就是立「技术栈与技术路线」基线的闸门卡，前端地基正是最不该事后返工的部分，应在此一次性夯实。

- **一套语言（TS/Node）**：画图库是 JS 的、最终在浏览器渲染，全栈一种语言最省心。
- **设计令牌单一来源**：ADR-0003 已用「圈」与阶段配色定义了视觉语义；把它落成令牌、图与 UI 共用，避免两处各调一套色导致不一致。
- **沿用 cytoscape + dagre**：已验证「真实数据画出来清爽好用、不是面条」（ADR-0003），无理由换库。
- **大陆网络**：原型期实测——运行时在线引 CDN 会白屏。改用 Vite 构建后，依赖在构建时打包成本地产物，**运行时完全不依赖外网**，比原型更稳；联网仅在 `npm install` 一次性发生（必要时走国内镜像）。

## 被否决的选项

- **沿用原型的「无框架 / 无构建 / 纯静态页」**——最薄，但视觉与组件结构无地基，日后必然为样式/组件化返工；与维护者「避免样式框架返工」的诉求冲突，故升级为 React + Tailwind。代价是 TB1 引入构建工具链、不再「双击即开」，维护者已知并接受这一取舍。
- **Python 作为主语言**——会与 JS 画图库、浏览器渲染割裂成两套语言。
- **TB1 直接上 Electron / Tauri / 常驻后端服务**——超出 TB1 范围；Vite 本地开发服务已够用，真正的常驻后端留到 Phase 2/3。
- **换 elkjs 等其它渲染/布局库**——cytoscape+dagre 已验证够用，换库是无收益的风险。
- **运行时 CDN 引入前端库 / 渲染库**——大陆会白屏（原型硬教训）；一律构建时本地打包。
- **依赖解析上 LLM**——约定已高度结构化，正则更准、可重复、可调试、零成本。

## 后果

- TB1 交付物 ≈「一个拉数据的 Node 脚本 + 一个 Vite/React/Tailwind 的本地 SPA」，纯只读（对齐 ADR-0001/0002 与 PRD #1）。比原型厚，但建立了可长期演进的前端地基。
- **设计令牌是后续所有视觉的单一来源**：#3（按阶段上色）、#4（就绪高亮）、#6（按 PRD 分组视图）、#7（开关/刷新）等卡的视觉一律从令牌取值，不得在组件里硬编码颜色。
- **纯逻辑模块（3/4/5）必须与 React/渲染解耦**：它们是「输入 → 输出」的纯函数，用 Vitest 单测，不依赖任何前端框架——PRD #1 的 6 模块拆分天然支持这一点，也为将来把「拉数据脚本」升级成本地服务降低迁移成本。
- **地基只建结构与约定，不铺满整套组件库**：TB1 只实现自己真正需要的组件（图容器、最小节点详情等），但 token 体系 / 组件分层约定 / 响应式断点要一次定好，让后续卡照此扩展而非返工。
- **运行时不依赖外网**；引入任何新前端依赖都要复核「大陆可直连 / 可走镜像」。
- 测试范式锁定为 Vitest + 行为测试，后续卡照此基线。
