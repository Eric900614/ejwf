# 驾驶舱长出「执行本机命令」的后端，跨进 Phase 2/3（派活）；但仍不写回 GitHub

## 决定

为了做 **派活**（半自动：从 **就绪** 卡一键拉起本机 agent 干活），驾驶舱**从纯前端只读 SPA，升级为带一个「能在本机执行命令」的本地后端**的应用。这个后端不是通用命令执行器，只暴露派活所需的 **Agent 启动器**：用 `execFile` + argv 数组 + `shell: false` 安全地拉起 allowlist 内的本机 agent CLI（CC / Codex）、喂提示词、捕获其输出、跟踪进程生命周期。

前端不得传任意命令、shell 片段或任意可执行文件路径；它只能提交「派哪张卡、用哪种 agent」这类领域输入。命令文件、工作目录、标准提示词文件路径由后端按当前 repo / 受控配置解析。提示词通过 agent CLI 支持的非交互参数或 stdin 注入，具体方式由 #29 / #30 spike 验证后固化。

这是产品**从「只读看得见」（TB1 + 看图 v2）正式跨进 Phase 2/3（能动手）**的里程碑。但**有一条边界不变**：驾驶舱自己**仍不写回 GitHub**（不改标签、不改正文、不开 PR）——动手改代码、提 PR 的是被拉起的 **Agent**，不是驾驶舱。**ADR-0001 的只读铁律（数据主权在 GitHub 标签 + 正文、驾驶舱不改）依然成立。**

## 背景 / 为什么

ADR-0002 早写明：「拉起本地 agent 是 Phase 2/3 的灵魂……托管网页碰不到用户的机器」，并为此选了「本地 app」。派活正是那条被故意挂起的 Phase 2/3 主线，是压低 **HITL 占比**（北极星）的大头。TB1 与看图 v2 把「看得见」做满后，维护者真受够了「手攒提示词 + 手敲启动命令 + 盯终端」的手动派活，决定啃它。

要拉起本机进程，托管网页 / 纯前端 SPA 做不到，必须有一个跑在本机、能执行命令的后端。ADR-0004 已为此预留落点（「Vite 本地开发服务天然是本地小服务，为将来 Phase 2/3 预留后端落点」）。本 ADR 把这个落点正式启用。

**关键区分——这一步破的是「纯只读 SPA 形态」，不是「只读数据」**：

- **破的**：驾驶舱不再只是静态前端，它长出了能在本机跑命令的「手」。
- **不破的**：驾驶舱对 GitHub 仍是只读（ADR-0001）。改代码 / 提 PR 是 agent 的行为，驾驶舱只负责「把 agent 连提示词一起拉起来」。

## 被否决的选项

- **继续纯只读、派活只做「最薄档」（拼提示词摆给维护者复制）**——零架构风险、不长手，但省不掉「手敲启动命令、盯终端」，离北极星半步之遥；维护者两次明确要「驾驶舱能一键拉起 agent」，故升级。
- **调云端 agent API（不拉本机进程）**——维护者实际用的是本机 CLI（终端跑 codex / claude），云端路线与现状不符，且 ADR-0002 的「本地 app」立意正是为了碰本机进程。
- **直接上 Electron / Tauri / 常驻后端服务**——超出第一刀；先用 Vite 已预留的后端落点 + spike 验证，重型常驻形态等真有需要再说。

## 后果

- 驾驶舱获得「执行本机命令」的能力——**安全责任随之上升**：后端只提供窄口径 Agent 启动器，不提供「运行任意命令」端点；一切外部输入（卡号、agent 类型、per-card brief 等）进入命令时一律走命令构造器输出的 `file + args + cwd + stdin` 结构，不做字符串插值，不启用 shell（沿用项目既有防注入约定）。
- **风险高一截**：本机无人值守拉起 agent，比纯前端渲染出错面大得多——故派活第一刀**以 2 个 spike（拉起 CC / 拉起 Codex）前置 de-risk**，验证「能无人值守拉起 + 能干活 + 不卡交互」才展开主体。
- **ADR-0001 / ADR-0006 不受影响**：驾驶舱仍不写 GitHub；私人标注仍本地存。本 ADR 只新增「执行本机命令」这一能力维度。
- **可迭代**：这是朝北极星的方向性赌注。手动派 → 半自动（本 ADR）→ 全自动，分阶段走；后端形态（Vite 中间件 / 独立 Node 服务 / 将来常驻）随 spike 与真实手感迭代。
