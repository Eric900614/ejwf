# 驾驶舱默认只显示 open 卡；已完成卡退为可选的灰色背景

## 决定

驾驶舱的**默认视图只渲染「活的卡」**——即 open 状态的 issue。已完成（closed）的卡
**默认隐藏**，仅作为可选的灰色背景，由一个开关按需调出（用来追溯「为什么链」与「已满足的依赖」，
对齐 PRD #1 的 User Story 15）。

随之确立两条配套表达：

- **首选视图是「按 PRD 分组」**（原型 B 版）：所有 open 卡按归属的 PRD 进泳道，无归属的进「散卡」道；
  「依赖图」（原型 A 版）为次选。两者都默认只画 open。
- **节点的依赖状态用「圈」表达**，因为隐藏 closed 后指向已完成前置的箭头会消失，需要补充信号：
  **金圈＝就绪可派**（`ready-for-agent` 且所有 blocker 已关）/ **绿圈＝依赖已就位**（前置都已完成、
  但未标 `ready-for-agent`）/ **有入箭头＝仍被未完成的前置挡着** / **无圈＝本无依赖**。
  UI 必须让「没箭头」可被解释（提示「前置多半已完成、已隐藏」），否则会被误读成 bug。

## 背景 / 为什么

一个真实 repo 会沉淀几百张卡，但**绝大多数是 closed**；当下真正要管理的 open 卡通常在 50 张以内。
若把全部卡都画上，视图会被「已经做完、不用再管」的卡淹没——这正是 ADR-0001 里那块死看板荒废的病灶之一
（「扁、装不下结构」叠加「噪声盖过信号」）。

原型用 `foravia` 真实数据实测：全量 59 节点（含被引用到的 23 张已关闭卡）/ 55 条边，**默认只留 open 后
降到 36 节点 / 22 条边**，可读性显著变好。用户（产品唯一用户）看图后确认：默认藏掉 closed、需要时再灰色展开，
正是让图「清爽好用、而非面条」的关键一招（PRD #1 User Story 16 的真未知，至此有了肯定答案）。

## 被否决的选项

- **默认渲染全部卡（open + closed）**——实测被完成项淹没，重蹈死看板覆辙。
- **完全不画 closed（连开关都不给）**——会丢掉「为什么链 / 已满足依赖」的来龙去脉，违背 US15；
  且「就绪」判定本就依赖「前置是否已关」，数据上仍需取已关闭的前置卡。故选「默认藏、可选灰显」。

## 后果

- 默认视图规模 ≈ open 卡数（通常 < 50），人眼可读；这是驾驶舱相对扁平 list / 死看板的核心体验优势。
- **GitHub 读取器（PRD 模块 1）仍须取「被引用到的已关闭卡」**：哪怕默认不显示，也要用它们判定 blocker
  是否已关、并在开关打开时画出「已满足」的依赖边。隐藏是渲染层的事，不是数据层少取。
- 因隐藏 closed 而消失的依赖箭头，须由上面的「圈」语义 + 开关 + 文案共同补偿，确保依赖状态不丢、不被误读。
- 「首选视图＝按 PRD 分组」是当前默认，非铁律；真实使用攒够手感后可再调（本 ADR 只锁「默认只盯 open」这一条硬决策）。
