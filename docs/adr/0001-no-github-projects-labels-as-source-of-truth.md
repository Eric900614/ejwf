# 驾驶舱不接 GitHub Projects；标签 + 正文为阶段与依赖的权威来源

## 决定

驾驶舱**不**把 GitHub Projects 看板当作数据主权层。**卡片的阶段**由 issue 的**标签**（+ 关联 PR 的状态）推导；**卡片之间的依赖**从 issue **正文里的 `#引用` 与关键词**（`Depends on`、`Source: ADR-xxxx`、`TB1/TB2`、`step 1/2`、`carve M1/M2`）解析。驾驶舱**自己渲染**视图（图状，而非扁平看板），TB1 只读取 GitHub 上已有的标签与正文；是否写回标签留到后续阶段单独决策。GitHub 那边的数据组织方式一个字都不用改。

## 背景 / 为什么

用户在另一个项目（`users/Eric900614/projects/2`）真跑过 GitHub Projects 看板，配了「按 tag 切泳道」的自动化后**仍然荒废**。荒废的三条原因，恰好就是本驾驶舱的核心卖点（反过来说）：

1. **看板是扁的，装不下结构** —— 真实工作是 `PRD → TB1/TB2/TB3`、`Depends on #690`、`ADR-0039` 这样一棵树 / 一张网；看板的列把它拍扁成「待办/进行中/完成」，把最关键的依赖、层级、顺序全丢了。
2. **它是给人看的，agent 不参与** —— 活是 agent 在 issue / PR / 标签上推的；看板是纯人工 UI，没人喂，自然脱节、废弃。
3. **只有「状态」，没有「为什么」** —— 工作围着 ADR / PRD 的推理链转，看板把这条「为什么」的线整个甩掉了。

## 被否决的选项

- **GitHub Projects v2 作为主看板** —— 已实测、已荒废，见上。其「按标签切泳道」的自动化反而证明了**标签才是阶段的真实信号**，看板只是标签的一个（很差的）投影。
- **改造工作流去用原生 sub-issues / 依赖字段** —— 要求改掉一套已经跑顺的习惯，且只覆盖新 issue；留作将来「驾驶舱反向自愈」的可选项，不作为起点。

## 后果

- 驾驶舱必须**自己实现**「图状视图」与「正文依赖解析」——这正是它相对 Projects 的价值所在，不是负担。
- 「阶段与标签同步」无需驾驶舱操心：GitHub 上标签是源头，驾驶舱读标签即可，不存在 Projects 时代的「双写、对不上」问题（用户确认该问题当年也不存在）。
