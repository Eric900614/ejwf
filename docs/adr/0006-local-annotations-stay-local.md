# 驾驶舱的私人人工标注纯本地存储，不写回 GitHub、也不进 repo

## 决定

在当前**纯只读驾驶舱**阶段，驾驶舱里**由人手动打、只服务个人看图体验的标注**——当前是 **盯 / 重点 (Pin)**——一律**只存在本机浏览器的 localStorage**（key 使用稳定仓库标识 `owner/repo` + issue 号）：**既不写回 GitHub、也不提交进 repo 文件**。它与 **就绪**、**龄期** 这类从 GitHub 数据**算出来的派生信号**是两类东西——私人人工标注是主观的、私人的，派生信号是客观的、可重算的。

若将来的分级优先级、备注或派活队列开始影响共享工作流、Agent 调度、验收放行等协作语义，它们就不再是本 ADR 范围内的「私人看图标注」，必须重新做架构决策，而不能直接套用 localStorage。

## 背景 / 为什么

本轮（看图体验 v2）的一条贯穿原则是**让驾驶舱保持纯只读、现在先不长出「写」的手**：

- **写回 GitHub** 会触碰 ADR-0001 的只读边界（卡片的阶段/依赖以 GitHub 标签+正文为权威，驾驶舱只读不改）。
- **写 repo 文件** 则需要驾驶舱具备「在本机写文件」的后端能力——这与「拉起本地 agent」是**同一类架构质变**（破「纯只读 SPA」形态，属故意挂起的 Phase 2/3）。

而「盯」只是维护者的私人偏好备忘，本就不该污染 GitHub 上的卡片数据、也不该提交进 repo。localStorage 纯前端即可，零架构风险，与当前只读 SPA 形态完全契合；key 用 `owner/repo + issue 号` 这样的稳定仓库标识，避免同名本地目录、不同 fork 或多 worktree 之间互相串数据。

## 被否决的选项

- **存进 repo 文件（如 `.cockpit/pins.json`，跟着 git 走）**——能跨机器同步、能版本化；但要驾驶舱长出「写文件的手」、破只读形态，且把私人偏好提交进 repo 不干净。为一个个人备忘标记，现在不值得跨这道坎。
- **写回 GitHub（issue 标签/正文）**——直接违 ADR-0001 只读铁律，且「盯」是私人视角，不该改动多方共享的 GitHub 数据。

## 后果

- 「盯」以及同类**私人、主观、不参与共享调度**的人工标注都走 localStorage；**代价是跨机器/跨浏览器不同步、清浏览器缓存会丢失**——已知并接受。驾驶舱本就是单机自用、本机常驻的本地 app（ADR-0002 已为此放弃「随处可开」），跨机器同步基本不是需求。
- 这是一条**朝「保持只读、不长手」方向的赌注**，与本轮「先做看图体验、把派活/长手能力推到 Phase 2/3」一致。
- **可迭代**：将来若产品整体迈进 Phase 2/3（驾驶舱已具备写本地 / 写回能力），可重新评估是否把人工标注升级为持久化 / 可同步存储。届时本 ADR 标 superseded 即可。
