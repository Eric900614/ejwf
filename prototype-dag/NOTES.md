## ⚠️ 原型期间抓到的一个真 bug（已修，值得记住）

第一版解析器用了正则 `\Z`（字符串末尾锚点）来界定 `## Blocked by` 段的结尾。
**但 JavaScript 正则不支持 `\Z`**——它被当成字母「Z」，于是凡是 `## Blocked by`
是正文最后一段的卡（绝大多数都是），整段匹配失败、blocked-by 边全丢。

后果触目惊心：边从 **55 → 17**，就绪从真实的 **16 → 虚高的 24**，孤立点 18 个，
图看起来「假性稀疏、假性清爽」。**若不是原型坚持喂真实数据，几乎必然得出「图很好读」的错误结论。**

修法：改成按行扫描段落（`## 标题` 下一行到下一个 `## ` 或文末），不依赖 `\Z`。
→ 给真产品模块 3（关系解析器）的测试用例留个教训：**`## Blocked by` 作为正文末段** 必须进测试覆盖。

## 用户反馈 → 设计洞察（2026-06-03，第一轮看图后）

- 用户一眼看穿：**v1 的 A/B/C 差别不大**（因为 B 只圈了 13 张、其余散着；C 仍留 41/59 节点）。属实，是 v1 没做好。
- 更关键的用户洞察：**一个项目大部分卡是 closed，真正要管的 open 卡其实 50 以内**——不该把已完成的绿点画满屏。
  → v2 据此重做：**默认只显示 open**（closed 退成可选灰背景），三版改成三种真正不同的「组织方式」：
  结构(A) / 按 PRD 归堆(B) / 派活看板(C)。这条「驾驶舱默认只盯活的卡」很可能要写进 ADR。

## 用户反馈 v2 → v3（2026-06-03，第二轮）

- 用户倾向 **B（按 PRD 分组）**，A 也认可；**C 砍掉**（看板方向不要了）。
- 用户在 B 里发现「框内缺连线、有的有有的没有」，觉得奇怪。**查实：不是 bug。**
  那些卡的 blocked-by 前置**全是已关闭卡**（如 #688←#684✅、#531←#527/528/529/530✅），
  默认隐藏 closed → 箭头连前置一起消失，看着光秃秃。而 slice-3 簇前置还开着 → 箭头都在。
  → v3 修法：① 砍 C，切换只剩 A/B；② 新增**绿圈**＝「前置都做完但未标 ready-for-agent」
  （#688/#687/#531），让「没箭头」不再含糊；③ 图例点明「没箭头≠没依赖，多半是前置已完成，
  勾『显示已关闭卡』看完整血脉」。
  分类（36 张 open）：金圈就绪 16 / 绿圈依赖已就位 3 / 仍被挡 9（有入箭头）/ 真没依赖 8。

## ✅ 结论（2026-06-03，用户看 v3 后拍板）

**原型验证通过 —— 可作为「验证核心功能的第一版」。** US16 那个真未知有答案了：

- **可读性**：真实数据（36 张 open）+ **默认只显示 open 卡** + 按 PRD 分组，**清爽好用**，
  不是面条。把已完成（closed）的卡默认藏起、需要时再灰色展开，是治「死看板看不清」的关键一招。
- **选定方向**：**B（按 PRD 分组）为主**，A（依赖图）次之；**C 看板砍掉**。
- **「就绪」高亮**：金圈一眼可见；外加绿圈（依赖已就位/非 agent 就绪）后，每张卡的依赖状态都不再含糊。

### 从原型里长出来的、值得固化的产品决策

1. **驾驶舱默认只显示 open 卡**，closed 退为可选的灰色背景（治死看板「被完成项淹没」之病）。
2. 节点依赖状态用「圈」表达：**金圈=就绪可派 / 绿圈=依赖已就位 / 有入箭头=仍被挡 / 无圈=本无依赖**。
3. 「没箭头」要可解释，否则用户会误以为是 bug（本轮真实踩到）。

### 给真产品（模块 3/4/5）留下的测试教训

- 关系解析器（模块 3）：**`## Blocked by` 作为正文末段** 必须进测试覆盖（本轮 `\Z` bug 的根源）；
  `## Parent` 段、多条 blocked-by、blocker 指向已关闭卡，都要覆盖。
- 阶段推导器（模块 4）：注意「就绪 ∩ 开发审查中」会重叠（同一卡既 ready-for-agent 又有 open PR），
  派活语义上应归「开发审查中」。
- 图构建+就绪（模块 5）：就绪 = ready-for-agent 且所有 blocker 已关；区分「依赖已就位」与「本无依赖」。
